## 結論先行

**Phase 3 の "mel-to-mel VC 近似" は根本的に誤っていた**。CosyVoice3 の DiT は raw mel を `mu` 入力として受け付ける設計ではなく、必ず **token encoder の隠れ状態** を `mu` として要求する。生 log-mel を流し込んだ結果、velocity 場が学習分布外で発散し、DiT 出力の **58% (4640/8000) が -inf** になっていた。これがノイズの根本原因。長さ 1/18 問題 (T=100 切断) はこれとは独立の構造問題で、両方を直す必要がある。

## 根本原因の特定 (reference 実装読解結果)

`C:/Users/tatuk/Documents/ComfyUI/custom_nodes/TTS-Audio-Suite/engines/cosyvoice/impl/cosyvoice/flow/flow.py` の `MaskedDiffWithXvec.inference()` (L102-145) を精読した結果:

```python
# CosyVoice3 reference 実装での DiT 入力構築
embedding = F.normalize(embedding, dim=1)
embedding = self.spk_embed_affine_layer(embedding)  # spks

h, _ = self.encoder(token, token_len)
h    = self.encoder_proj(h)                          # ← mu の元

conds                = torch.zeros([1, mel_len, 80])
conds[:, :mel_len1] = prompt_feat                   # prompt あり時のみ非ゼロ
conds                = conds.transpose(1, 2)        # ← cond

feat = self.decoder(
    mu   = h.transpose(1, 2),    # encoder hidden state
    spks = embedding,            # 投影済 speaker emb
    cond = conds,                # zeros (prompt 部分のみ非ゼロ)
    ...)
```

### C# との差分

| 入力 | reference | 私たち (C#) | 影響 |
|------|-----------|------------|------|
| `mu` | encoder + encoder_proj の隠れ状態（おおむね tanh 範囲） | **生 acoustic mel** `[-11.5, +2.7]` | 学習分布外 → 発散 |
| `cond` | zeros テンソル (VC prompt 無し時) | **mu と同じ source mel** | 別経路でも分布外入力 |
| `spks` | L2 norm → affine projection | 192→80 線形投影 (近似) | △ 軽微 |
| `t_scheduler` | **cosine** (`1 - cos(t·π/2)`) | 線形 (uniform) | 微妙にズレる |
| CFG | `inference_cfg_rate=0.7`, batch=2 でガイダンス | 無し (export 時に baked-in 想定) | 要 export 時挙動確認 |

**Phase 3 のテスト T1-T5 が pass したのは shape と数値有限性しか検証していなかったため**。聴感品質の検証が抜けていた。

### 関連
- mel 抽出の Hann window symmetric / n_fft 2048 ゼロパディング問題は **DiT 発散の前段で副次的**。修正しても根本問題は解決しない
- 1785 frames の `source_mel.npy` は matcha の数値比較対象として引き続き有用

## 注意点・ブロッカー

- **Phase 3 の `feat-vc` ブランチで Pass した T1-T5 は聴感品質を検証していない**。次セッションで T7 (E2E 聴感テスト) を追加すべき
- **mu の値域不一致は致命的**: 学習時 `tanh` 範囲 vs 実行時 log-mel 範囲で 5〜10 倍スケール違い。スケーリングだけで誤魔化せる問題ではなく、本質的に別の信号
- **CFG (inference_cfg_rate=0.7) の export 時挙動が未確認**: ONNX export スクリプトでバッチ次元をどう扱ったかで挙動が変わる。🅱 を進める時に要再確認
- **t_scheduler='cosine'** vs C# の uniform は副次的な誤差源。🅱 で encoder 統合する際に同時修正
- **MelExtractor の uncommitted 変更**は 🅰 検証で活きる（matcha 一致を進めるべき）。Hann window の symmetric → periodic 修正と n_fft=1920 のままの FFT 化が次に効く
- 現在 `.claude/worktrees/` と `voiceCoppy_test/diag_*.py` (15 個) が untracked
