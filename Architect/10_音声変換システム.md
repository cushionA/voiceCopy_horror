# 10. 音声変換システム (Voice Conversion)

> voice_horror プロジェクトの公式アーキテクチャ章。
> Phase 7 (kNN-VC 統合) 実装着手前に成立した技術選定とクラス設計を記録する。
> 詳細仕様は `designs/specs/voice_conversion/{spec,design,tasks}.md` を参照。

## 採用技術

| 要素 | 採用 | 役割 |
|------|------|------|
| **kNN-VC** (bshall/knn-vc, MIT) | コア変換アルゴリズム | フレーム単位の特徴 kNN 検索 |
| **WavLM Large** (Microsoft, MIT) | 自己教師あり特徴抽出器 | 音声 → 1024 次元特徴ベクトル × N frames、VC + 類似度判定で兼用 |
| **HiFiGAN** (kNN-VC 同梱、MIT) | Vocoder | 特徴ベクトル → audio 波形 |
| **Unity Inference Engine (Sentis 2.5)** | ML ランタイム | ONNX 推論、`com.unity.ai.inference: 2.5.0` |

合計 VRAM: ~750-900MB (CosyVoice3 系の 1/3)。

## なぜ kNN-VC か (撤退判断との比較)

`docs/reports/handoffs/2026-05-09_phase3.5-pivot-to-knn-vc.md` 詳細。要約:

| 観点 | CosyVoice3 (撤退) | kNN-VC (採用) |
|------|-----------|----------------|
| アーキテクチャ | LLM token decoder + DiT + flow + vocoder (4 モデル) | WavLM + kNN + HiFiGAN (2 モデル) |
| VRAM | ~2.6GB | ~900MB |
| Sentis 互換性 | 7-8 種類の op patch / merge / value_info ストリップ等 | パッチなしで動作 (Phase 6 検証済) |
| 学習要件 | zero-shot (LLM 事前学習済) | zero-shot (forward pass のみ) |
| narrative 適合 | 単発 zero-shot 変換 | プール蓄積で「徐々に混ざる」演出が原理的一致 |
| 類似度判定 | 別途モデル (ECAPA-TDNN 等) 要 | WavLM 兼用 (1 モデル 2 役) |

## システム全体図

```
[マイク入力 / AudioClip]
        │
        ▼
WavLMFeatureExtractor (Sentis Worker, GPUCompute)
        │  16kHz mono → [N, 1024] features (WavLM Layer 6)
        │
   ┌────┴────┬───────────────────┐
   ▼         ▼                   ▼
TargetPool  PlayerPool      query (一時)
(永続)     (蓄積、永続)
   │         │                   │
   └────┬────┘                   │
        ▼                        │
   WeightedPoolBuilder           │
   (α 比率で重み構築)             │
        │                        │
        └──→ [N_total, 1024] features + weights [N_total]
                          │
                          ▼ ← query
                  KnnVcConverter (CPU/SIMD)
                  (重みつき kNN, k=4)
                          │
                          ▼ converted features [N_query, 1024]
                  HiFiGANVocoder (Sentis Worker)
                          │
                          ▼ audio (16kHz mono)
                  AudioClip 出力 + peak 正規化
                          │
                          ▼
                  AudioSource で再生
```

別経路で **SpeakerSimilarityJudge** が TargetPool と PlayerPool 全体の cosine 類似度を
算出し、エンディング分岐 (BAD「奪われる」/ BAD「電話切れ」/ GOOD) を判定する。

## クラス分割と責務

| クラス | 責務 | 実装場所 |
|--------|------|---------|
| `WavLMFeatureExtractor` | audio → 1024-dim 特徴抽出 (Sentis Worker wrapper) | `Assets/SentisSpike/Scripts/KnnVc/` |
| `MatchingSetPool` | 特徴ベクトルの蓄積、weight 管理、npy 永続化 | 同上 |
| `WeightedPoolBuilder` | 2 プール + α → 連結プール + weights 配列 | 同上 |
| `KnnVcConverter` | 重みつき kNN 検索、cosine_dist 計算 | 同上 |
| `HiFiGANVocoder` | features → audio 復元 (Sentis Worker wrapper) | 同上 |
| `SpeakerSimilarityJudge` | プール間コサイン類似度 + ED 分岐 | 同上 |
| `EdThresholdsAsset` | ED 判定閾値 (ScriptableObject、hot-fix 可) | 同上 |
| `VoiceConversionService` | ファサード、MonoBehaviour、DontDestroyOnLoad | 同上 |

詳細 API は `designs/specs/voice_conversion/design.md` 参照。

## 「混ざっていく」演出の実装メカニズム (narrative 整合)

voice_horror のコアメカニクス「少女幽霊の声に徐々にプレイヤーの声が混ざっていく」は、
**WeightedPoolBuilder の α 値時間変動**で実装する:

```
ゲーム序盤  : α = 1.0  (純少女声)
ゲーム中盤  : α = 0.7  (少女 7 割 + プレイヤー 3 割)
ゲーム終盤  : α = 0.3  (少女 3 割 + プレイヤー 7 割)
ED 直前   : α = 動的 (プレイヤー声蓄積量に応じて)
```

α は `VoiceConversionService.Convert(source, targetWeightAlpha)` の引数として渡し、
内部で `WeightedPoolBuilder.Build(TargetPool, PlayerPool, alpha)` → `KnnVcConverter.Convert(query, mergedPool, weights)` の経路で実現される。

## エンディング分岐ロジック

ゲーム終了時、`VoiceConversionService.JudgeEnding()` が以下を実行:

1. PlayerPool 全体の平均特徴 = `[1024]` ベクトル A を計算
2. TargetPool 全体の平均特徴 = `[1024]` ベクトル B を計算
3. `sim = max(0, cos(A, B))` (clamp 必須、SS-001)
4. `EdThresholdsAsset` に従い 3 分岐:
   - sim > High (デフォルト 0.85) → **BadCaptured** 「カラダを奪われる」
   - sim > Low (デフォルト 0.40) → **BadVoiceChange** 「声色を変えて攻略 → 電話切れ」
   - sim ≤ Low → **Good** 「脱出成功」

閾値は仮の値で、Phase 8 で 5 名内部キャリブレーションで実測調整する。

## メモリ予算 (Sentis ランタイム)

| 項目 | サイズ |
|------|--------|
| WavLM Large (FP16) | ~600MB |
| HiFiGAN | ~50MB |
| TargetPool (5 分音声、15000 frames × 1024 × 4 bytes) | ~60MB |
| PlayerPool (累積、想定 5 分上限) | ~60MB |
| Sentis ランタイム + 中間テンソル | ~150MB |
| **合計** | **~920MB** |

Unity URP + UHFPS の ~5GB と合わせて ~6GB、RTX 2070S 8GB に対して 25% 余裕。

## レイテンシ予算 (Phase 6 実測ベース、warmup 後)

| ステージ | 1 秒音声あたり |
|---------|--------------|
| AudioClip → 16kHz リサンプル (CPU) | < 5ms |
| WavLM forward (Sentis GPUCompute) | ~8ms |
| kNN match (CPU/SIMD、~15000 frames target) | ~5-15ms |
| HiFiGAN vocode (Sentis GPUCompute) | ~12ms |
| 16kHz → AudioClip (CPU) | < 5ms |
| **合計** | **~35-45ms** (RTF 0.04, 22-28x real-time) |

**初回 forward (cold start)** は ~510ms かかるため、シーン遷移 (タイトル → ゲーム開始) で
warmup 1 回実行して隠す (SR-002)。

## 永続化

### TargetPool

声優収録音声 (Phase 9) を Python 側で WavLM forward → npy 化したものを `StreamingAssets/voice_horror/pools/girl_ghost.npy` (+ `_weights.npy`) として配布。
ゲーム起動時に Service.Initialize() でロード。

### PlayerPool

ゲームプレイ中の録音 (チュートリアルマイクテスト、ジャンプスケア時の悲鳴等) を逐次 forward → append。
`Application.persistentDataPath/voice_horror/pools/player.npy` (+ `_weights.npy`) に保存。
セーブ/ロードで復元。
