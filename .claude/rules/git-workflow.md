# Git 運用ルール

## ブランチ戦略

```
main              ← 安定版（リリース可能な状態を維持）
├── feature/○○   ← 機能開発
├── bugfix/○○    ← バグ修正
├── hotfix/○○    ← 緊急修正（mainから分岐、mainへマージ）
└── refactor/○○  ← 大規模リファクタリング
```

### ブランチ命名規則

```bash
# 機能開発
feature/プレイヤー移動実装
feature/インベントリシステム

# バグ修正
bugfix/カメラ揺れ修正
bugfix/HP表示ずれ修正

# 緊急修正
hotfix/v1.0.1

# リファクタリング
refactor/入力システム刷新
```

- ブランチ名は日本語OK（このプロジェクトの方針）
- 短く、何の作業か分かる名前にする
- feature ブランチは main から分岐し、main へマージ
- **区切りついたらPRでmainに統合し、ブランチは適切に切り替える**（長期ブランチを避ける）

## コミットルール

### コミットメッセージ形式

```
[種類](範囲): 日本語タイトル

- 変更内容の箇条書き（任意）

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
```

### コミットタイプ

| 種類 | 用途 | 例 |
|------|------|-----|
| feat | 新機能追加 | feat(player): ジャンプ機能を実装 |
| fix | バグ修正 | fix(camera): 追従時の揺れを修正 |
| test | テスト追加・修正 | test(combat): ダメージ計算テスト追加 |
| refactor | リファクタリング | refactor(input): InputHandler分離 |
| style | コードスタイル修正 | style(全体): 命名規則統一 |
| perf | パフォーマンス改善 | perf(enemy): GetComponent呼び出しキャッシュ化 |
| chore | その他の作業 | chore(config): editorconfig追加 |
| asset | アセット追加・更新 | asset(sprite): プレイヤースプライト追加 |
| docs | ドキュメント | docs(設計): GDD更新 |

### スコープ例

| スコープ | 対象 |
|----------|------|
| player | プレイヤー関連 |
| enemy | 敵関連 |
| ui | UI関連 |
| audio | オーディオ関連 |
| scene | シーン関連 |
| system | システム全体 |
| stage | ステージ関連 |
| event | イベントシーン関連 |
| pipeline | パイプライン関連 |

### コミットタイミング

パイプライン実行時の自動コミットポイント:
1. **機能実装完了時**: テスト全Pass後に `feat(scope): 機能名を実装` でコミット
2. **設計書作成時**: GDD/システム設計書生成後に `docs(設計): セクション名の設計完了` でコミット
3. **ステージ構築時**: ステージシーン構築後に `feat(stage): ステージ名を構築` でコミット

手動作業時:
- 論理的な単位で小さくコミット（1機能 = 1コミットが目安）
- 動かないコードをコミットしない
- テストとそのテスト対象の実装は同じコミットに含める

## プッシュルール

- **コミット後は必ずプッシュする**（バックアップ + 進捗共有）
- パイプライン実行中: 各機能実装完了時にプッシュ
- force-push は原則禁止（`--force-with-lease` のみ許可、rebase後に限る）

## Unity固有のGitルール

### .meta ファイル管理
- アセット追加時は `.meta` ファイルも必ず一緒にコミット
- アセット削除時も `.meta` ファイルの削除を確認
- `.meta` ファイルだけが残る/消えることがないようにする

### シーン・プレハブの扱い
- シーン変更後は必ず Unity で保存してからコミット
- コミット前チェック:
  - [ ] シーンが保存されている
  - [ ] コンソールにエラーがない
  - [ ] プレイモードで基本動作が確認できる

### ステージング禁止対象
- **アセットストア由来など、他者に権利があるアセットはステージしない**
- 機密情報（APIキー、credentials）は絶対にステージ・コミットしない
- 有料アセットや第三者ライセンスのファイルはリポジトリに含めない

### .gitignore
- `Library/`, `Temp/`, `Logs/`, `obj/` は必ず除外
- ビルド出力 (`Build/`, `Builds/`) は除外
- IDE設定 (`.vs/`, `.idea/`) は除外

- `Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>` を付与

## プルリクエストルール

PR作成時は以下の手順で進める:

1. **テスト確認**: 全テストがPassしていることを確認する
2. **PR作成**: `gh pr create` でPRを作成する
   - タイトル: 日本語、70文字以内
   - 本文: Summary（変更概要）+ Test plan（テスト計画）を記載
3. **コードレビュー実行**: PR作成後にPRの差分を分析し、`gh pr review` でレビューコメントを付ける
   - Code Reuse: 重複パターン、既存ユーティリティ未使用
   - Code Quality: バグ、命名規約違反、イベントリーク、リソースリーク
   - Efficiency: ホットパスアロケーション、キャッシュ漏れ、不要なsqrt

## バージョニング（必要時）

セマンティックバージョニング: `v大.中.小`
- **大 (Major)**: 大幅な変更・互換性なし
- **中 (Minor)**: 機能追加（後方互換あり）
- **小 (Patch)**: バグ修正
