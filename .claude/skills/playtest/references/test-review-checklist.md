# テストレビュー専門家 — テスト設計の十分性判断

## 役割
テスト専門家（t*.md）が設計したテストと、機能の実装コードを見比べて、テストが十分かを判断する。
不足があれば具体的な不足箇所を指摘し、再設計を要求する。

## レビュープロセス

### 入力
- 機能の実装コード（Step 1 で収集済み）
- 各テスト専門家が設計したテスト（Step 2 の出力）

### レビュー手順

#### 1. 公開API網羅性
実装ファイルの全publicメソッドに対して、テスト設計がカバーしているか確認する。

- [ ] 全publicメソッドに最低1つのテストが対応しているか
- [ ] メソッドの戻り値/副作用の両方が検証されているか

**不足時の指摘例**: 「`ApplyDamage()` に対するT1テストがない」

#### 2. 分岐条件網羅性
実装コード内の条件分岐（if/switch/三項演算子/早期return）を列挙し、テストがカバーしているか確認する。

- [ ] 各分岐の真/偽の両方にテストがあるか
- [ ] 早期returnの条件がテストされているか（例: `if (!IsAlive) return;`）
- [ ] switch文の全caseにテストがあるか（default含む）

**不足時の指摘例**: 「`IsAlive == false` の早期returnパスがT1でテストされていない」

#### 3. 境界値網羅性
数値パラメータ・距離判定・タイミング判定の境界値がテストされているか確認する。

- [ ] 数値の上限/下限/ゼロ/負数がテストされているか
- [ ] 距離閾値の「ちょうど内側」「ちょうど外側」がT7でテストされているか
- [ ] タイミング閾値（コンボ受付時間等）の境界がT2でテストされているか

**不足時の指摘例**: 「攻撃範囲 = 2.0f だが、T7で距離2.0と2.1のテストがない」

#### 4. 状態遷移網羅性
状態を持つ機能（ステートマシン、フラグ、カウンタ等）の遷移パスがテストされているか確認する。

- [ ] 正常遷移パス（A→B→C）がテストされているか
- [ ] 逆遷移（C→B→A）がテストされているか（可能な場合）
- [ ] 同一状態への再突入（A→A）がテストされているか
- [ ] 無効な遷移が拒否されることがテストされているか

**不足時の指摘例**: 「攻撃中→攻撃の再突入（コンボ中断）がT2でテストされていない」

#### 5. 副作用・連携検証
機能が他システムに与える副作用（イベント発火、SoAデータ書き込み、コンポーネント状態変更）がテストされているか確認する。

- [ ] イベント発火が検証されているか（T1またはT3）
- [ ] SoAデータへの書き込みがT3で検証されているか
- [ ] 他コンポーネントのenable/disable変更がテストされているか
- [ ] リソース確保/解放の対称性がテストされているか

**不足時の指摘例**: 「`OnDamageReceived` イベントの発火がどのテストでも検証されていない」

#### 6. エラーパス・防御的コード検証
防御的コード（null チェック、IsAlive チェック、範囲チェック等）が実際に機能することがテストされているか確認する。

- [ ] nullチェックに対して、null入力時の動作テストがあるか
- [ ] 死亡キャラクターへの操作テストがあるか（T7 + T3）
- [ ] 無効化オブジェクトへの操作テストがあるか（T7）

**不足時の指摘例**: 「死亡後の被弾処理（HitBox の IsAlive チェック）が T7 でテストされていない」

## 判定基準

### 十分（続行）
- 上記6カテゴリの全チェックを確認済み
- 機能の複雑度に見合ったテスト数がある
  - 単純機能（分岐2-3個）: 最低5テスト
  - 中程度機能（分岐5-10個）: 最低10テスト
  - 複雑機能（分岐10個以上）: 最低15テスト

### 不足（再設計要求）
- 上記チェックで未カバーの項目がある
- 具体的な不足箇所をテストタイプ（T1-T9）と共に指摘
- 不足の重要度を判定:
  - **必須不足**: ◎テストタイプでの不足 → 必ず再設計
  - **推奨不足**: ○テストタイプでの不足 → 可能なら再設計
  - **軽微不足**: △テストタイプでの不足 → 記録のみ、続行可

### レビューループ上限
- 最大2回。2回のレビューで収束しなければ現状の設計で進行
- レポートに「レビュー未収束: [不足箇所]」として記載
