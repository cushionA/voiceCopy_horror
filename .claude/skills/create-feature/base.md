# create-feature base — HOW (プロジェクト非依存の TDD ワークフロー)

> このファイルは `.claude/skills/_two-layer-design.md` で定義された Two-layer 構成の **base 層**。
> SisterGame 固有の配置・参照 rules・コマンドは overlay (`SKILL.md`) を参照する。
> 本ファイルは TDD サイクル・git ワークフロー・自己レビュー観点など「他プロジェクトに移植可能な手順」のみを記述する。

---

## 0. ブランチ作成（汎用 git workflow）

新機能実装前にメインブランチから新しい feature ブランチを切る。

```bash
git checkout <main-branch>
git pull
git checkout -b <feature-branch-name>
```

- ブランチ命名は overlay の規約に従う
- 既に他の feature ブランチ上にいる場合はユーザーに確認:
  「現在 `<current-branch>` にいます。新しいブランチを切りますか？」
- ユーザーが明示的にブランチ不要と言った場合のみスキップ

## TDD サイクル（Red → Green → Refactor）

### ステップ 1: 仕様確認

機能の入出力、責務範囲、依存関係を整理する。
overlay が指定する仕様フォーマット（テンプレート）に基づいて文書化する。

### ステップ 2: テスト作成（Red）

テストを**先に**作成する。
- 機能を分解した複数の検証ケースをテストとして記述
- 命名は **「[機能名]_[条件]_[期待結果]」**（具体例は overlay 参照）
- ファイル配置先は overlay が指定する

### ステップ 3: Red 確認

テストが**全て Fail する**ことを確認する。
- すでに Pass するテストがあれば、それは新機能ではなく既存機能の検証になっている可能性がある
- 全件 Fail でない場合はテストの前提条件を見直す

### ステップ 4: 実装（Green）

テストを Pass させる**最小限の実装**を作る。
- 過剰実装を避ける（YAGNI）
- 既存コードとの整合性を確認する
- overlay が指定する規約・テンプレート参照ルールに従う

### ステップ 5: Green 確認

テストが**全て Pass する**ことを確認する。
- ランタイム環境のエラー / 警告も確認する（overlay が具体的なログ確認手段を指定）

### ステップ 5.5: 結合テスト（必要に応じて）

単体テストだけでは検証できない**システム間の繋がり**をテストする。
具体的な観点とテンプレートは overlay の rules ファイルが定義する。

結合テストが不要なケース:
- 純粋なデータ構造体・enum 等
- 他システムへの依存がない独立ロジック

### ステップ 6: 自己レビュー

実装完了後、以下の言語非依存な観点で自己チェック:

- [ ] 既存コードとの重複はないか（DRY）
- [ ] 公開インタフェースは最小限か（カプセル化）
- [ ] マジックナンバーが裸で書かれていないか（定数化）
- [ ] **既存ユーティリティを正しく経由しているか**（既存ロジック直書きでバイパスしていないか）
- [ ] **イベント購読と解除が対称か**（subscribe / unsubscribe、register / dispose）
- [ ] **リソース確保と解放が対になっているか**（ハンドル、メモリ、外部接続）
- [ ] エラーハンドリングが boundary で行われているか（内部で過剰防御していないか）

言語固有の規約チェックリストは overlay の rules ファイルが追加する。

---

## ステップ 7: 記録

完了した機能を機能 DB に記録する。
overlay が具体的なコマンドを指定する。

**記録するメタデータ**:
- 機能名
- テストファイルパス
- 実装ファイルパス
- ステータス（complete / failed / in_progress）
- テスト件数（passed / failed）

将来タスクから着手したケースでは、overlay の指示に従ってタスクトラッカー側にも完了マークを入れる。

---

## ステップ 8: Git コミット

テスト全 Pass + 記録完了後にコミット & プッシュ。

- 該当 feature ブランチにいることを確認
- テストファイルと実装ファイルを `git add`
- コミットメッセージは overlay が指定する書式に従う
- プッシュ: `git push origin <branch>`

---

## エラー時のリトライ

- テストが Fail し続ける場合、修正を試みる（最大 3 回程度）
- それを超える場合はユーザーに状況を報告し、判断を仰ぐ
- スコープが膨らみすぎた場合は機能分割を提案する

---

## I/O コントラクト（base 層が要求する overlay 提供物）

overlay 側で必ず指定すべき項目:

1. **仕様フォーマット** — 例: `instruction-formats/feature-spec.md`
2. **テストファイル配置パス** — 例: `Tests/EditMode/{Name}Tests.cs`
3. **テスト実行手段** — 例: MCP `run_tests`、Unity CLI バッチ
4. **コード規約 rules ファイル** — 例: `.claude/rules/unity-conventions.md`
5. **結合テスト rules ファイル** — 例: `.claude/rules/test-driven.md` の「結合テスト」節
6. **機能 DB 操作コマンド** — 例: `python tools/feature-db.py add/update`
7. **ブランチ命名規則** — 例: `feature/<機能名>`
8. **コミットメッセージ書式** — 例: `feat(scope): 機能名を実装` + `Co-Authored-By: Claude Opus 4.7`
9. **将来タスクトラッカー** — 例: `docs/FUTURE_TASKS.md` の更新ルール
10. **アセット要求の登録先**（必要に応じて）— 例: feature-db `add-asset` コマンド

これらが全て揃わないと base のロジックは動かない。overlay でこれらを束ねる責務を持つ。
